# Phase 2.5b — Agent Onboarding UX

## Status: Draft — pending user review

## Context

CertOps agent onboarding must produce a provisioned agent that Phase 2.5c
(billing/limits) and Phase 2.5d (enterprise dashboard) can build against
without guessing at its shape. This spec covers the onboarding flow,
credential handoff, agent identity contract, and telemetry push integration.

This design depends on and stays consistent with:
- **Phase 1** (closed): telemetry contract + scoped agent auth + repo split
- **Phase 2** (closed): multi-tenant dashboard + read isolation
- **Phase 2.5a** (closed): connector registry + env-var auto-detection + DB seeding

## Decisions (locked)

- **Operator**: Human, interactive CLI wizard. Unattended/fleet provisioning
  is explicitly out of scope for this phase.
- **Token flow**: Wizard calls dashboard registration API → dashboard issues
  scoped agent token → token stored locally in `agent.db`.
- **Wizard scope**: Full setup in one command — dashboard registration,
  secret store detection/credentials, local persistence.
- **Enterprise infra credentials**: Out of scope. Only secret store credentials
  are collected; host connector credentials deferred to a later phase.
- **Agent storage**: Separate local SQLite database (`agent.db`), distinct from
  dashboard's `certops.db`.
- **Exit criterion**: Working end-to-end wizard + documented, testable definition
  of "what a provisioned agent is" that 2.5c/2.5d can design against.

## 1. Agent Identity Layer (Dashboard-side)

### New: `agents` table

```sql
CREATE TABLE IF NOT EXISTS agents (
    id TEXT PRIMARY KEY,           -- uuid4 hex, assigned at registration
    tenant_id TEXT NOT NULL,
    name TEXT,                      -- operator-provided agent name (optional)
    status TEXT NOT NULL DEFAULT 'pending',
    -- Status values: pending | registered | configured | active
    registered_at TEXT NOT NULL,
    configured_at TEXT,
    activated_at TEXT,
    last_seen_at TEXT,
    secret_store_backend TEXT       -- vault | azure_kv | null
);
```

### Modified: `agent_tokens` table

Add `agent_id` column to link tokens to agents:

```sql
ALTER TABLE agent_tokens ADD COLUMN agent_id TEXT;
-- Foreign key: agents(id)
```

### New: Dashboard API endpoints

| Endpoint | Method | Auth | Purpose |
|----------|--------|------|---------|
| `POST /api/agents/register` | POST | Admin bearer | Create agent + token |
| `GET /api/agents` | GET | Admin bearer | List agents for tenant |
| `GET /api/agents/{id}` | GET | Admin bearer | Get agent details |

### Registration flow

1. Wizard sends `POST /api/agents/register` with admin bearer token
2. Dashboard creates `agents` row (status: `pending`)
3. Dashboard creates scoped token in `agent_tokens` (with `agent_id` FK, `scope: "telemetry_push"`)
4. Dashboard returns `{ agent_id, tenant_id, token }`
5. Wizard stores these in `agent.db`

### Registration API contract

**Request:**
```json
POST /api/agents/register
Authorization: Bearer <admin_token>

{
    "name": "prod-agent-01"    // optional
}
```

**Response:**
```json
{
    "agent_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "tenant_id": "default",
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "status": "pending"
}
```

## 2. Agent Local DB (`agent.db`)

Separate SQLite database, stored alongside the agent (default: `./agent.db`).

### Schema

```sql
CREATE TABLE IF NOT EXISTS agent_identity (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Keys: agent_id, tenant_id, token, dashboard_url,
--        secret_store_backend, status, registered_at,
--        configured_at, activated_at

CREATE TABLE IF NOT EXISTS agent_config (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Keys: vault_addr, vault_token, azure_keyvault_url,
--        azure_tenant_id, azure_client_id, azure_client_secret,
--        renewal_threshold_days, step_ca_url, etc.
-- Values encrypted with CERTOPS_CONFIG_ENCRYPTION_KEY
```

### Why key-value?

- Simpler to extend (add new fields without migrations)
- Matches the `.env` pattern but with DB-backed state
- Status field drives the resumable wizard

### Resumability

The wizard reads `agent_identity.status` on startup:

| Status | Wizard behavior |
|--------|----------------|
| `pending` | Start from step 1 (dashboard registration) |
| `registered` | Skip to step 2 (secret store config) |
| `configured` | Skip to step 3 (validation cycle) |
| `active` | "Agent is already set up. Run `certops agent status`." |

### Secret store detection (consistent with 2.5a)

The wizard prompts for connection details (endpoint + credentials), then
uses the 2.5a detection logic to validate and identify the backend:

1. User enters connection details (endpoint URL, token/creds)
2. Wizard hands those details to the 2.5a detection logic
3. Detection validates connectivity and identifies the backend
4. Wizard confirms detected backend to the user ("Detected: HashiCorp Vault at ...")

If the user hasn't entered details yet, the wizard offers a menu of
backends to guide them. The menu is a UX convenience, not a backend
selection — the actual identification happens via detection logic.

### Implementation

- New CLI subcommand in `certops-agent/src/cli.py`
- Each step is a function that can be called independently
- State transitions written to `agent.db` after each step
- Uses `input()` for prompts (no new dependencies)

## 4. Telemetry Push Integration

### Current state

`AgentTelemetryClient` exists in `agent_telemetry.py` but is never called
from `run_renewal_loop()`.

### Change

Wire telemetry push into the renewal loop:

```python
# In run_renewal_loop(), after each renewal cycle:
if agent_id and agent_token and dashboard_url:
    client = AgentTelemetryClient(
        agent_id=agent_id,
        agent_version=VERSION,
        agent_token=agent_token,
        ingest_url=f"{dashboard_url}/api/telemetry/ingest"
    )
    client.push(summary_payload)
```

### Agent.db reads on startup

`run_renewal_loop()` reads from `agent.db` (if it exists) to get:
- `agent_id`, `token`, `dashboard_url` for telemetry push
- `secret_store_backend` to skip env-var seeding if already configured

If `agent.db` doesn't exist (legacy mode), the agent runs as before —
local-only, no telemetry.

## 5. Secret-Blindness Guarantee

### Rules

1. Dashboard admin credentials: held in memory only during the registration
   API call, never persisted to disk or logged
2. Secret store credentials (Vault token, Azure creds): stored only in
   `agent.db` (encrypted), never sent to dashboard
3. Dashboard receives: `agent_id`, `tenant_id`, `secret_store_backend` (type
   only, not credentials), telemetry data
4. Dashboard never receives: Vault token, Azure client secret, SSH keys,
   WinRM passwords

### Test assertion

An explicit test must assert that no credential value appears in any
payload sent to the dashboard API during the full wizard flow.

## 7. Out of Scope

- Host/enterprise infra connector credentials (Nginx/IIS targets)
- Non-interactive/unattended provisioning as a documented mode
- Billing/limit enforcement (2.5c)
- Dashboard UI for agent management (2.5d)
- Agent token rotation or expiry enforcement
