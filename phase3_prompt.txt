## Phase 3.0 — Append-Only Audit Log

### Interface contract
- New table `renewal_log`: `id` (PK, autoincrement), `cert_id` (FK), `timestamp`, `event_type` (`discovered` / `renewal_started` / `renewed` / `deployed_pending_reload` / `reload_confirmed` / `reload_failed` / `error`), `connector_category`, `connector_type`, `old_expiry`, `new_expiry`, `old_fingerprint`, `new_fingerprint`, `success` (bool), `detail` (text).
- No `UPDATE` or `DELETE` statement against `renewal_log` may exist anywhere in the codebase. This is a hard rule, not a convention — grep for it in your own smoke test.
- Implement logging as a **decorator or context manager** wrapping each connector's state-transition functions, not manual `insert_log_entry()` calls scattered inline — the guarantee we want is that a log row is written even when the wrapped function raises. Manual calls at each call site risk a code path that renews but forgets to log on the exception branch.
- Wire this into all four existing connectors (`HashiCorpVaultClient`, `AzureKeyVaultClient`, `SSHHostConnector`, `WinRMHostConnector`) at every state transition each already passes through.

### Out of scope this phase
- No frontend work.
- No retention/pruning policy for the log table — it grows unbounded for now.
- No changes to `pipeline_stage` semantics — this phase only adds logging alongside it.

### Smoke test (hard exit criteria)
- Run a real renewal through at least one secret-store connector and one host connector (reuse existing verified paths: HashiCorp Vault → step-ca → Nginx; SSH host → step-ca → SFTP deploy → confirmed reload).
- Force one failure (e.g., kill step-ca mid-renewal, or point at a bad SSH port) and confirm a `success=false` row with a real error in `detail` is still written — this is the actual point of the phase, so don't skip it.
- Query and paste raw `SELECT * FROM renewal_log ORDER BY id` output showing both successful and failed rows.
- Grep the codebase for `UPDATE renewal_log` and `DELETE FROM renewal_log` and paste the (empty) output confirming neither exists.

## Phase 3.1 — Groups

### Interface contract
- New table `group`: `id`, `name`, `description`.
- New join table or FK column on `cert` (`group_id`) — decide which based on whether a cert can belong to more than one group; ask Arpit this explicitly if it's not already resolved from earlier design discussion. ### Out of scope this phase

- No notification logic.
- No frontend.
- No default/"ungrouped" auto-assignment logic beyond a nullable `group_id` — a cert with no group is just null, not auto-bucketed.

### Smoke test

- Assign existing test certs across both groups.
- Query `get_due_certificates()` (or equivalent) filtered by group and paste raw output showing correct filtering.
- Confirm certs with no group assignment still surface correctly in the unfiltered due-list query (no regression).

## Phase 3.2 — Maintenance Windows

### Interface contract
- New table `maintenance_window`: `id`, `group_id` (FK), `start_time`, `end_time`, `recurrence` (keep this simple — a single field like `once` / `weekly`, not a full cron/RRULE implementation unless Arpit asks for it explicitly).
- Renewal/deploy/reload logic must check: is this cert's group inside an active maintenance window right now? If not, the pipeline holds at whatever stage it's naturally gated at (this dovetails with the existing host-connector "pending reload" gate — maintenance windows become another reason a reload waits, alongside the existing manual confirm gate).
- **Explicit design question to ask Arpit before coding:** does a maintenance window gate the *renewal* (cert fetch from CA) too, or only the *deploy/reload* step? Given the pull-model / zero-key-exposure design, renewal itself is probably safe to run anytime — it's the live service reload that's disruptive. Confirm this assumption before implementing rather than guessing.

### Out of scope this phase
- No notification tie-in yet.
- No recurring-window edge case handling beyond the simple recurrence field above (no timezone-DST edge cases, no overlapping-window conflict resolution) — flag these as known gaps in your report rather than silently handling them with added complexity.

### Smoke test
- Create a maintenance window for a group that is currently **closed** (not active) and confirm a due renewal for a cert in that group is correctly held/blocked at the gate you implemented — paste raw log/DB output showing the hold.
- Create a window that is currently **open** and confirm the same cert proceeds — paste raw output showing successful completion through to the stage the gate controls.

## Phase 3.3 — Notification Policy + Notification Log

### Interface contract
- New table `notification_policy`: `id`, `group_id` (FK), `threshold_days` (an int; supports multiple rows per group for stacked thresholds, e.g. 45-day heads-up and a separate 30-day one).
- New table `notification_log`: `id`, `cert_id`, `policy_id`, `sent_at`. This table's job is purely deduplication — before sending, check whether a `notification_log` row already exists for this `(cert_id, policy_id)` pair; if so, don't resend.
- Notification firing is **independent of auto-renewal** — a 45-day notice must fire even though it's nowhere near the renewal threshold. Implement this as its own check pass, not bolted onto `get_due_certificates()`.
- Actual notification *delivery* mechanism: ask Arpit what he wants for now — a stub (log to console/DB only) is acceptable and probably correct for this phase; don't build real email/Slack integration unless explicitly asked.

### Out of scope this phase
- No real email/SMS/Slack delivery unless Arpit confirms he wants it now.
- No notification *content* templating system — a plain string is fine.

### Smoke test
- Set a policy with a threshold that a test cert should currently trigger.
- Run the check pass twice in a row and confirm via raw `SELECT * FROM notification_log` output that the second run does **not** produce a duplicate row — this is the actual point of the phase.
- Confirm a cert's renewal is unaffected by (does not consume/clear) the notification firing — the two systems stay decoupled.

## Phase 3.4 — Scheduler (RenewalJob, event-driven not polling)

### Interface contract
- Add `next_renewal_at` (and, if useful, `next_notification_check_at`) to the cert row, computed as `expiry - renewal_threshold_days` at write/renewal time.
- Replace any tight-loop/fixed-interval polling with per-cert scheduled wake-ups. Given the earlier discussion, **default to Temporal** (durable per-cert workflow that sleeps until `next_renewal_at`, renews, redeploys, re-arms itself off the new expiry) over Celery+Redis, since Temporal's durable timers avoid needing a separate DB-resync sweep job on restart. If Arpit wants Celery+Redis instead (already has infra provisioned, wants less new infra to learn), confirm that explicitly before building — this is a real decision point, not a default to silently override.
- On restart, the scheduler must recover in-flight and future-scheduled state from the DB without missing a cert — this is the actual hard requirement, independent of which scheduler tech is chosen.

### Out of scope this phase
- No UI for viewing/managing scheduled jobs — that's Phase 3.6 (frontend).
- No manual job cancellation/retry UI — CLI/DB-level intervention is fine for now.

### Smoke test
- Schedule a cert due in ~60 seconds, kill the scheduler process, restart it, and confirm (raw output) the job still fires at the correct time — this is the actual point of the phase, so don't skip the restart step.
- Confirm a cert with a `next_renewal_at` far in the future produces zero CPU/API activity in the interim (no polling) — show this via logs, not just an assertion.
- Note explicitly in your report: since test certs still use step-ca's short lifetime, `next_renewal_at` will compute to "almost immediately" for test certs and to realistic future dates once DigiCert is wired in. This is expected, not a bug — confirm your report states this rather than treating short-lived test scheduling as a defect.
